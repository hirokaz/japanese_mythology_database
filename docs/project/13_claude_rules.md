# Claude 運用ルール — 暴走防止規程

Claude が **長時間自律実行** する前提で、暴走・断定・偏重・幻覚生成・スコープ逸脱を防ぐためのルール。CLAUDE.md §4 を補強する実装レベルの規程。

---

## 1. 絶対禁止事項(違反時はその場でコミット中止)

### 1.1 仮説断定禁止

- **L0(史料記載)以外を「である」と書かない**
- 中世神道書・地方伝承・近代国学は L0 ではない
- 「学説」「通説」も L1-L2 → 必ず「とされる」「と考えられている」

### 1.2 出典なし記述禁止

- TSV 各行は **必ず `出典` 列を埋める**(空欄禁止 → やむを得ない場合は「不明」)
- master TSV の主要属性(canonical_name, parent_deity_id 等)も出典に基づく
- relation の `source_reference` も必須

### 1.3 神話成立年代と神話描写時代の混同禁止

- 古事記成立(712 年)≠ 古事記が描く神代
- 中世神道書(『神道五部書』『中臣祓訓解』等)は **中世編集** 視点で扱う
- 鎌倉成立の説を「上代の神話」と書かない

### 1.4 myth ↔ history の境界混同禁止

- 神武〜開化(欠史八代)を歴史実在と書かない
- 神話登場神(イザナギ・大国主)を歴史人物として記述しない
- 神功皇后の三韓征伐を「史実」と書かない(L1-L2 仮説)

### 1.5 中央史観の絶対化禁止

- 古事記・日本書紀の語りを地方伝承の上位に置かない
- 「正しい神話」として朝廷編纂版を扱わない
- 出雲国風土記独自エピソードを「異伝」と矮小化しない

### 1.6 スコープ逸脱禁止

- 1 Issue で複数 Epic にまたがる変更を行わない
- 「ついでに」リファクタを行わない
- 別 Issue として起票してから着手

### 1.7 master_id の捏造禁止

- 存在しない master_id を relation の source/target に記載しない
- 不明な場合は **新規 master 起票 Issue** を別途立てる

### 1.8 一括上書き禁止

- 既存 master TSV を **全面再生成しない**(差分更新)
- relation TSV も同様
- 既存行の改変は理由を commit message に記載

## 2. 必須プラクティス

### 2.1 Read-before-Edit の徹底

- 編集前に **当該ファイル全行を Read**
- 関連スキーマ(`docs/schema/01_*` 〜 `09_*`)を再確認
- CLAUDE.md §3 の 29 列定義を再確認

### 2.2 出典の格付け

各記述には以下のレベル付けを必ず:

| レベル | 内容 | 表記 |
|---|---|---|
| 一次史料 | 古事記/書紀/風土記/延喜式/万葉集 | 「○○ 巻第 N」 |
| 一次史料(社伝) | 由緒・縁起 | 「○○神社縁起」 |
| 二次史料 | 中世神道書 | 「神道五部書 ○○」 |
| 系譜資料 | 新撰姓氏録/先代旧事本紀 | 「新撰姓氏録 ○○段」 |
| 近代研究 | 学術論文 | 「著者名(年)」 |
| 民俗・伝承 | 地方伝承 | 「○○地方伝承」 |

### 2.3 confidence_level / hypothesis_layer の付与

- relation 全件に必須
- L4-L5 → confidence_level は必ず E
- L0 → confidence_level は通常 A-B
- 不一致を疑う場合は監査 issue 起票

### 2.4 myth / history / local_tradition の分離

- TSV の `分類` カラムで明確化
- 必要なら別ファイル(`<編>/05_myth.tsv` / `<編>/06_history.tsv`)に分離

### 2.5 speculative flag の付与

- 自明でない仮説は `仮説レベル` カラムで L3 以上を付与
- 大胆仮説(L4-L5)は **独立 hypothesis ノード** に分離して関連 supports/contradicts で結ぶ

## 3. 自律実行モードでの追加ルール

Claude が `claude-autonomous` ラベル付き Issue を実行する場合の追加制約。

### 3.1 セッション境界

- 1 セッション = **1 Issue** に限定(複数 Issue 横断禁止)
- セッション開始時に Issue 本文を **Read** で全文確認
- 完了時にセッションログを Issue にコメント

### 3.2 Subtask 完了ごとのチェックポイント

各 Subtask 完了時:

- [ ] 該当 TSV/md を保存
- [ ] git diff を確認(意図しない変更がないか)
- [ ] Issue にチェック✓を更新
- [ ] 次 Subtask へ

### 3.3 ブロック時の対応

- 不明エンティティ・出典不確定に当たったら **`risk:source-weak` ラベル** を付け、当該行に `史実性レベル: D` または `E` を付与
- 完全に判断不能なら Issue を `status:needs-info` で停止し、人間判断を待つ

### 3.4 PR セルフレビュー

PR 作成後、Claude は以下を **必ず実行**:

```markdown
## セルフレビュー

- [ ] CLAUDE.md §3.2 の 29 列を満たしている
- [ ] 全行に「出典」記載
- [ ] L4-L5 仮説を断定していない
- [ ] master_id 重複なし
- [ ] relation の source/target が master に存在
- [ ] 表記ゆれ検査(自分の編内で同一エンティティ複数表記がない)
- [ ] 神話成立年代と神話描写時代を分離
- [ ] 中央史観に偏っていない
- [ ] スコープ内に留まっている
- [ ] PR diff のサイズが妥当
```

→ 全項目 ✓ なら Approve & Merge。1 つでも × なら修正してから再レビュー。

## 4. 「神話を否定しない」原則

CLAUDE.md §4 の原則を実装レベルで:

| やってよいこと | やってはいけないこと |
|---|---|
| 神話を **L0(史料記載)** として整理 | 神話を「ただの作り話」として切り捨て |
| 神話の **政治的読解** を hypothesis ノード化 | 政治的読解を史実扱い |
| 神話を **考古学と対比** | 考古学が無いことを根拠に神話を全否定 |
| 縄文記憶の可能性を **L4-L5 仮説** で残す | L4-L5 を断定 |

---

## 5. 「地方を中央に従属させない」原則

| やってよいこと | やってはいけないこと |
|---|---|
| 出雲国造神賀詞を **独立 text master** として扱う | 古事記の補足として扱う |
| 諏訪のミシャグジを **独立 deity** として扱う | タケミナカタの「地方異伝」として吸収 |
| 阿蘇の健磐龍命を **独立 deity** として扱う | 神武天皇の系譜の一部として吸収 |
| 風土記独自エピソードに **独立 myth_episode** を割り当て | 中央神話の異伝としてのみ記述 |

---

## 6. 「仮説を断定しない」原則の具体実装

### 6.1 NG 表現と OK 表現

| NG(断定) | OK(層別化) |
|---|---|
| 「銅鐸祭祀の終焉は国譲り神話に対応する」 | 「銅鐸祭祀の終焉が国譲り神話に対応するという L1 仮説がある」 |
| 「ヤマタノオロチは製鉄民の隠喩である」 | 「ヤマタノオロチを製鉄民の隠喩とする L4 仮説がある」 |
| 「アラハバキは縄文神である」 | 「アラハバキは縄文系祭祀の残存とする L4 仮説がある」 |
| 「卑弥呼は天照大神である」 | 「卑弥呼=天照大神説は L4 大胆仮説」 |

### 6.2 hypothesis ノードの作り方

L4-L5 を扱うときは必ず:

```
HYP-XXX  hypothesis_master に登録
   │
   ├── supports → 支持事実(SITE/ARTIFACT/MYTH)
   ├── contradicts → 反証事実
   └── proposed_by → 提唱者(TEXT/CLAN/EMPEROR)
```

事実 node 側に「○○仮説では…」と書き込まない。**仮説は仮説 node に集約**。

## 7. 並行作業時の衝突回避

複数 Claude セッションが並行する場合:

### 7.1 ファイル分離

- **異なる編** を担当:衝突なし(別ディレクトリ)
- **同じ編の異なるサブトピック**:`<編>/04_*.tsv` と `<編>/05_*.tsv` のように分離
- **同じ master**:同時編集禁止(`status:in-progress` で排他)

### 7.2 ブランチ命名

- `issue/ISE-04`(Issue ID 完全一致 → 衝突しない)
- 並行作業ブランチは `claude/parallel-work-<random>`(本ブランチがその例)

### 7.3 master 統合は順次

- 各地域編の master 抽出は並行可
- **統合 Issue(MST-DE-17 等)は 1 つずつ順次**

## 8. ハルシネーション対策

### 8.1 不明 → 「不明」と書く

CLAUDE.md §4 通り、不明事項を捏造しない。

### 8.2 引用箇所の特定

- 「日本書紀」→ NG。「日本書紀 神代上 第七段一書(三)」のように **巻段引用** を必須化
- 「古事記」→ NG。「古事記 中巻 神武天皇段」 等

### 8.3 数値の根拠

- 神社数(例: 全式内社 2861 社)、遺跡年代などの **具体数値** には必ず出典
- 「約 5000 社」のような曖昧表現は許容するが、出典は必須

### 8.4 人名の存在確認

- 中世社家・神職名などは確認の取れないものを書かない
- 確認困難な場合は省略 or `不明` で済ませる

---

## 9. 修正サイクル

### 9.1 Gemini 監査で指摘が入った場合

1. 該当行を **特定**
2. 修正案を **書き出し**(Issue コメント)
3. 修正 PR を作成
4. Gemini 再監査
5. OK ならマージ

### 9.2 Claude セルフ監査で問題発見

1. 即座に修正 PR 起票
2. 監査ログを `docs/audit/` に追記
3. schema 改訂が必要なら別 Issue 起票

---

## 10. 「やってよいこと」チェックリスト(セッション開始時)

```markdown
- [ ] このセッションは 1 つの Issue に紐づいているか
- [ ] Issue 本文を全文 Read したか
- [ ] 関連スキーマ・CLAUDE.md を確認したか
- [ ] 既存の関連 TSV を Read したか
- [ ] 着手ブランチが正しいか (issue/<Issue ID>)
- [ ] スコープ内に留まる確信があるか
```

すべて Yes でなければ着手前に確認すること。
